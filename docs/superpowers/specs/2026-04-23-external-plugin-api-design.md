# External Plugin API — Design

**Date:** 2026-04-23
**Status:** Approved

---

## Goal

Allow plugins written in any language (Python, Go, Rust, etc.) to publish notifications to Arrr via HTTP, flowing through the existing `IEventBus` pipeline to all subscribers (Unix socket, D-Bus).

---

## Architecture

No new logic in `Arrr.Core` beyond a request DTO. All routing lives in `Arrr.Service`, following the existing `IHttpCallbackPlugin` pattern.

```
Arrr.Core/
└── Data/
    └── Api/
        └── ExternalNotifyRequest.cs     ← request DTO

Arrr.Service/
└── Api/
    └── ExternalNotifyEndpoint.cs        ← MapExternalApi() extension method
```

Modified files:

| File | Change |
|---|---|
| `Arrr.Core/Data/Config/ArrrConfig.cs` | add `ApiKey` property (default `""` = endpoint disabled) |
| `Arrr.Core/Data/Config/ArrrConfigJsonContext.cs` | register `ExternalNotifyRequest` |
| `Arrr.Service/Program.cs` | call `app.MapExternalApi()` |

Flow:

```
POST /api/notify
    → ExternalNotifyEndpoint
        → validate X-Api-Key header
        → validate required fields
        → build Notification (Id = Guid.NewGuid(), Timestamp = UtcNow)
        → IEventBus.PublishAsync<Notification>
            → SocketBroadcastSubscriber → /tmp/arrr.sock
            → DBusNotifySubscriber     → org.freedesktop.Notifications
```

The two existing subscriber channels receive external notifications exactly like in-process plugin notifications.

## API Contract

```
POST /api/notify
Header: X-Api-Key: <key>
Content-Type: application/json
```

Request body:

| Field | Type | Required |
|---|---|---|
| `source` | `string` | yes |
| `title` | `string` | yes |
| `body` | `string` | yes |
| `iconUrl` | `string?` | no |

`Id` and `Timestamp` are always auto-generated by Arrr.

Response codes:

| Code | When |
|---|---|
| `204 No Content` | notification published successfully |
| `400 Bad Request` | `source`, `title`, or `body` missing or empty |
| `401 Unauthorized` | `X-Api-Key` header absent or wrong |
| `503 Service Unavailable` | `apiKey` not configured in `arrr.config` |

---

## Configuration

```json
{
  "apiKey": "my-secret-key"
}
```

If `apiKey` is `""` (default), the endpoint responds `503` until the user configures a key. This prevents accidental open endpoints.

`ArrrConfig.cs` addition:

```csharp
public string ApiKey { get; set; } = "";
```

## ExternalNotifyEndpoint

```csharp
// Arrr.Service/Api/ExternalNotifyEndpoint.cs
namespace Arrr.Service.Api;

internal static class ExternalNotifyEndpoint
{
    public static IEndpointRouteBuilder MapExternalApi(
        this IEndpointRouteBuilder app,
        IConfigService configService,
        IEventBus eventBus)
    {
        app.MapPost("/api/notify", async (HttpContext ctx, ExternalNotifyRequest req) =>
        {
            var key = configService.Config.ApiKey;

            if (key == "")
                return Results.Problem("API key not configured", statusCode: 503);

            if (!ctx.Request.Headers.TryGetValue("X-Api-Key", out var provided) || provided != key)
                return Results.Unauthorized();

            if (string.IsNullOrWhiteSpace(req.Source) ||
                string.IsNullOrWhiteSpace(req.Title)  ||
                string.IsNullOrWhiteSpace(req.Body))
                return Results.BadRequest("source, title and body are required");

            var notification = new Notification(
                Guid.NewGuid(),
                req.Source,
                req.Title,
                req.Body,
                DateTimeOffset.UtcNow,
                req.IconUrl
            );

            await eventBus.PublishAsync(notification);
            return Results.NoContent();
        });

        return app;
    }
}
```

## Error Handling

- API key check runs before body validation (fail fast).
- `PublishAsync` does not throw toward the endpoint; the event bus logs handler errors internally.
- Unhandled exceptions fall through to ASP.NET Core's default `500` response.
